GitHub announced major upgrades to their Pages site yesterday.  I woke up, read the news, and immediately created a new project site in six minutes.   I recorded my first impressions of the tool.



The automatic page generator is a great way to get the site started from a template.  Nice features include:

  • Nifty button to fork the repository;
  • Easy download of .zip and .tgz archives of the repository;
  • Attractive banner, title, and subhead style automatically created;
  • Easy import of README.md


I was able to add in images of the applications running.  However, I had to edit the markdown language and then find the URL for the graphic file.  This might take a bit of documentation if you have your marketing team promoting the project site.



After the project site was created, I created a Google Analytics ID and inserted the tracking code directly into the HTML.  To accomplish this, the marketing/social media team may need some help to avoid dropping the code snippet into the wrong place in HTML document.

I researched this process on Google and it seems that you'll need to access the gh-pages branch from within GitHub, not GitHub Pages.



Select the gh-pages branch.


select index.html.


Once you open index.html, click on the Edit button.  When you click on the Edit button, you'll see the code for editing.  Note that the default background is black, not gray.  If it's not black, make sure you clicked on the Edit button.


At this stage, you can edit the HTML code, your scripts, or drop in the Google Analytics code snippet.

The unfortunate part of this process is that whoever is updating the project site is going to need to know at least a minimal of HTML.


An example campaign using GitHub Pages might involve these steps:
  • Set up repository to gather code examples from the community.  For example, if you want to have people use your SDK, then you can solicit code examples and accept them with GitHub;
  • Seed the repository with example code from your own staff;
  • Set up GitHub Pages project site with both technical and marketing information;
  • Track  GitHub Pages with Google Analytics or other marketing automation tool;
  • Set up ongoing PR campaign to drive traffic to the project site;
  • Set up ongoing social media program to provide updates to code examples, buzz;
  • Highlight great code and people using blogs and social media.
Kunalo can handle all of these steps for you as part of a developer relations in a box campaign.

GitHub Pages are relatively new and right now the tool doesn't bring new functionality to marketing campaigns.  For example, there's limited social interaction between the marketing GitHub Pages site and the GitHub repository site.  The analytics are  handled by Google Analytics, nothing new.  Although you can gain the same functionality on your corporate site and roll the marketing analytics into your existing marketing automation package, I feel that GitHub Pages fills a useful niche.

GitHub Pages offer a fast and easy way to create software development project sites that offer community-contributed code.  Due to the ease of creating the site and value of the GitHub brands, I'll give this a thumbs up for community-contributed code.   Hopefully, this will fall under the same, fairly open, corporate usage guidelines as a Facebook Page, Twitter Feed, corporate blog, or YouTube channel.  

In order to attract more developers to an SDK, library, or platform, we're going to need to see more tools to organize, collect, and distribute code examples.  We're also going to need to integrate this process with the marketing promotion of the code examples.  This is a great early step and I encourage GitHub to keep adding tools for marketers.
